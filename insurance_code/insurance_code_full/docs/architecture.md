## 1. 架构目标与范围

### 1.1 目标
- 建设一个基于微信生态的保险数字化经营平台，覆盖 C/B/P 三端。
- 支持“内容学习 + 活动积分 + 商城兑换 + 保单托管 + 顾问服务”闭环。
- 支持多租户 SaaS，满足租户隔离、角色权限、审计合规要求。

### 1.2 范围划分
- C端（客户）：互动门户、学习、活动、保障管理、个人中心。
- B端（业务员）：客户经营工作台、内容与活动运营、核销、保单服务、数据看板。
- P端（平台/租户管理）：租户与组织管理、模板配置、策略引擎、平台运维与数据看板。

### 1.3 当前实施策略（按业务决策）
- C端功能契约先冻结（v1）。
- B端、P端先按最小可用能力上线，允许增量迭代。

## 2. 关键架构原则

- 契约优先：前后端先冻结 API Contract 再并行开发。
- 多租户隔离优先：所有业务数据强制 `tenant_id` 作用域。
- 安全合规优先：手机号等敏感数据加密存储，审计日志不可篡改。
- 事件可追踪：积分、核销、活动、学习全部有流水和幂等键。
- 渐进演进：先单体模块化，后按域拆分服务。

## 3. 总体架构

### 3.1 逻辑分层
- 表现层：C端 H5、B端 H5、小程序/APP 预留、P端 Web。
- 接入层：API Gateway（鉴权、限流、租户上下文注入、审计打点）。
- 业务层（领域服务）：
  - 用户与认证域（Auth）
  - 客户与关系域（CRM）
  - 内容学习域（Learning）
  - 活动与任务域（Campaign）
  - 积分与权益域（Points）
  - 商城与订单域（Mall）
  - 核销域（Writeoff）
  - 保单托管域（Policy）
  - 组织与权限域（IAM）
  - 配置与装修域（Config）
  - 分析与看板域（Analytics）
- 数据层：MySQL/PostgreSQL（OLTP）、Redis（缓存与锁）、对象存储（媒体）、消息队列（异步任务）。

### 3.2 物理部署建议（生产）
- Web/H5：Nginx + CDN（静态资源）
- API：容器化（K8s/ECS）
- DB：主从 + 自动备份
- Redis：主从哨兵
- MQ：Kafka/RabbitMQ（二选一）
- 对象存储：OSS/COS/S3
- 监控：Prometheus + Grafana + Loki + APM

## 4. 领域边界与职责

### 4.1 C端核心域
- Auth：实名（姓名+手机号+验证码）与会话。
- Learning：课程分类、课程详情、学习完成。
- Campaign：任务展示、签到、任务完成。
- Points：积分余额、流水。
- Mall/Writeoff：兑换、核销码、核销状态。
- Policy：保单列表、详情、OCR上传。

### 4.2 B端核心域
- CRM：客户列表/筛选、客户全景档案、互动轨迹。
- ContentOps：课程/活动/商城货架配置与排序。
- ServiceOps：客户兑换核销、订单查看。
- PolicyOps：保单录入、OCR复核、服务入口。

### 4.3 P端核心域
- TenantOps：租户、套餐、组织、成员、角色。
- TemplateOps：平台模板维护、租户启用与二次配置。
- StrategyOps：标签规则、策略工作流。
- PlatformOps：监控、对账、全局分析。

## 5. 多租户与权限架构

### 5.1 租户模型
- 强制字段：`tenant_id`、`org_id`、`team_id`、`owner_user_id`。
- 所有查询通过中间件注入租户条件。
- 跨租户访问默认禁止。

### 5.2 RBAC + ABAC 组合
- RBAC：平台管理员、公司管理员、团队主管、业务员、普通员工。
- ABAC：按数据归属（owner/team/tenant）、审批状态、有效期附加控制。

### 5.3 特殊访问流程
- 公司管理员查看客户明细：审批流 + 临时授权（时效、范围）+ 审计。
- 导出客户数据：二次验证 + 全量审计。

## 6. 数据模型（核心实体）

- `users`：用户主表（C/B/P）
- `user_verify_basic`：基础实名记录（姓名、手机号密文、验证状态）
- `tenants` / `org_units` / `memberships` / `roles`
- `customers`：客户主档（归属业务员）
- `courses` / `course_progress` / `course_completions`
- `activities` / `activity_tasks` / `activity_participations`
- `point_accounts` / `point_transactions`
- `mall_items` / `redemption_orders` / `writeoff_records`
- `policies` / `policy_members` / `policy_upload_jobs`
- `content_assets` / `content_publish_records`
- `audit_logs` / `approval_flows` / `idempotent_records`

关键约束：
- 积分流水唯一幂等键：`(user_id, source_type, source_id, biz_date)`
- 核销码唯一且一次性：`writeoff_token UNIQUE` + `status lock`
- 软删除：关键业务表 `is_deleted` + `deleted_at`

## 7. API 架构与契约策略

### 7.1 路由命名建议
- C端：`/api/c/*`
- B端：`/api/b/*`
- P端：`/api/p/*`
- 管理通用：`/api/admin/*`

### 7.2 版本策略
- C端先冻结 `v1`，只做向后兼容变更。
- B/P端走 `v1.x` 迭代，可新增接口与字段。
- 破坏性变更必须升级 `v2`。

### 7.3 错误码规范
- 业务冲突：`409`
- 参数错误：`400`
- 未认证：`401`
- 无权限：`403`
- 未找到：`404`
- 服务异常：`500/504`

## 8. 关键业务链路设计

### 8.1 实名拦截链路
1. C端触发签到/兑换/领奖。
2. 网关校验 `is_verified_basic`。
3. 未实名则返回拦截码，前端弹实名。
4. 实名成功后回跳并重放原操作。

### 8.2 积分发放链路
1. 行为上报（签到/课程完成/任务完成）。
2. 幂等校验（防重复）。
3. 事务内更新：积分账户 + 积分流水 + 行为状态。
4. 异步事件推送（看板、通知）。

### 8.3 兑换核销链路
1. 兑换时事务扣积分与库存，生成核销令牌。
2. 核销时服务端行锁校验 token 状态。
3. 成功后令牌置已用，写核销流水。
4. 日终核销对账。

### 8.4 保单托管链路
1. 客户/B端发起上传（手动或OCR）。
2. OCR结果入临时表并可人工修正。
3. 审核通过写正式保单表并建立客户关联。
4. C端/B端按权限查看保单详情。

## 9. 非功能架构设计

### 9.1 性能
- C端首屏 P95 <= 1.5s
- 核心接口 P95 <= 800ms
- 关键积分/核销接口 P95 <= 500ms

### 9.2 可用性
- 核心服务可用性 >= 99.9%
- 关键链路成功率 >= 99.5%

### 9.3 安全
- 手机号等敏感字段加密（AES-256 或国密）
- 日志脱敏
- 高危操作审批 + 审计
- 防重放、防暴力破解、防刷频（验证码、签到、兑换）

### 9.4 可观测
- 统一 TraceID
- 结构化日志
- SLI/SLO + 告警
- 业务埋点（view/click/sign_in/learn/redeem/write_off）

## 10. 技术选型建议

### 10.1 后端
- 推荐：`Node.js + NestJS` 或 `Java Spring Boot`
- 当前已有 Node/Express 原型，可平滑迁移至 NestJS 模块化。

### 10.2 前端
- C端/B端：React + Vite（先 H5）
- P端：React + Ant Design Pro（管理后台效率高）
- 小程序/APP：后续可用 Taro/uni-app 或 RN/Flutter 演进。

### 10.3 数据与中间件
- DB：PostgreSQL/MySQL
- 缓存：Redis
- 队列：Kafka/RabbitMQ
- 搜索（可选）：Elasticsearch（客户检索、内容检索）

## 11. 分阶段实施计划

### Phase 1（当前）C端闭环 + 基础后台
- C端：实名、学习、活动、积分、商城、核销、保单上传。
- B/P端：最小运营能力（活动、商品、内容上架/排序）。
- 交付：可运营演示版本。

### Phase 2（B端强化）
- 客户全景档案、轨迹、标签规则、保单服务流程。
- 团队主管视图与核销工作台。

### Phase 3（P端强化）
- 租户体系、组织权限、模板中心、策略引擎、审批审计。
- 公司管理员装修与模板二次配置。

### Phase 4（数据与智能）
- 全域分析看板
- 策略自动化
- 风险预警与客户意向评分

## 12. 风险与对策

- 风险：C/B/P权限混用导致数据越权。  
  对策：网关租户注入 + 统一权限中间件 + 审批流。
- 风险：积分重复发放与核销重复。  
  对策：幂等键、唯一索引、分布式锁、审计补偿工具。
- 风险：后续三端并行导致契约漂移。  
  对策：OpenAPI 为单一事实来源，变更评审机制。
- 风险：原型快迭代影响排期。  
  对策：冻结 C端 v1，B/P 增量迭代。

## 13. 当前需确认的架构决策

1. 后端最终框架：继续 Express 模块化，还是迁移 NestJS。  
2. DB 选型：PostgreSQL 或 MySQL（建议 PostgreSQL）。  
3. 队列选型：Kafka 或 RabbitMQ。  
4. 审批流是否首期上线（建议至少保留表结构和接口壳）。  
5. 小程序/APP 计划时间点（影响接口形态和鉴权策略）。
