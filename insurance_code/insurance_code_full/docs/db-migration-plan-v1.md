# insurance_code 数据迁移计划（v1）

更新时间：2026-02-23
迁移目标：将现有 `db.json` 迁移到 PostgreSQL，并保证C端业务不停摆。

## 1. 迁移策略决策

- 采用“分阶段迁移 + 短期双写 + 灰度切流”。
- 迁移工具：Prisma Migrate（结构）+ Node脚本（历史数据导入）。
- 原则：先兼容、后切换、最后下线`db.json`。

取舍理由：
- 一次性切换风险高；双写可降低回滚成本。
- Prisma迁移可审计、可复现，适合2周交付。

## 2. 分阶段执行

### Phase 0（D1）准备
1. 创建数据库与账号、最小权限。
2. 初始化迁移目录，生成v1基线DDL。
3. 在staging先跑一次全量迁移演练。

### Phase 1（D2-D3）历史数据导入
1. 从`db.json`导出用户、活动、积分、商城、兑换、课程、保单。
2. 执行字段清洗：时间格式、枚举合法性、空值兜底。
3. 导入后执行核对SQL（数量、金额、余额一致性）。

### Phase 2（D4-D6）双写与读切换
1. 写链路双写（PostgreSQL主写，db.json兼容写）。
2. 读链路按模块灰度切到PostgreSQL（活动->积分->商城->保单）。
3. 每模块灰度后观察24小时指标。

### Phase 3（D7-D8）停用db.json
1. 关闭db.json写入，保留只读回滚窗口48小时。
2. 完成最终一致性核对。
3. 删除对db.json的运行时依赖。

## 3. 校验清单（发布门禁）

- 用户数一致（容差0）。
- 积分账户余额 = 流水净额（容差0）。
- 每条兑换订单均可关联到商品与用户。
- 保单详情可正确聚合责任和缴费记录。
- OpenAPI契约回归通过（字段与语义不变）。

## 4. 回滚预案

- 应用层：回滚到上一版本镜像。
- 数据层：
  1. 启用只读模式。
  2. 恢复迁移前快照到旁路实例。
  3. 切回旧实例连接。
- 回滚触发条件：
  - 核心接口错误率 > 5% 持续5分钟
  - 积分余额不一致样本 >= 3例
  - 兑换核销出现重复成功

## 5. 实施责任建议

- 迁移脚本与DDL：后端负责人（R）
- 数据校验与抽样：QA + 数据工程（R）
- 发布与恢复演练：DevOps（R）
- 变更审批：架构师/技术负责人（A）
