## 2. 目标

本版不是冒烟清单，而是完整回归基线，覆盖：

1. 从平台管理员新建租户开始。
2. 在租户下创建团队、员工、页面权限。
3. 使用新建员工登录 B 端，测试其可见模块与业务功能。
4. 使用真实客户在 C 端实名，串起学习、活动、积分商城、保障、个人中心、分享。
5. 回流验证 B 端效果数据、P 端规则/指标/对账/权限隔离。
6. 验证模板从 `平台 -> 租户管理员 -> 员工 -> 客户` 的逐级可见、默认失效、手动激活后再下发。

## 3. 重要说明

### 3.1 Midscene 编写原则

- 所有 UI 操作按 `aiAct()` 分阶段描述。
- 一次 `aiAct()` 只做一个页面阶段，不跨多页长链路。
- UI 结果必须至少配一层接口或数据侧复核。
- `积分/指标/标签` 不接受只看页面渲染，必须使用“基线值 -> 执行动作 -> 增量值”的差值法验证。
- 对于当前 UI 没有独立入口的步骤，允许使用 API 作为前置数据准备。

### 3.2 当前已知 UI 边界

- 当前 P 端侧边栏没有独立的“客户分配”页面入口。
- 客户归属链路应通过 API `POST /api/p/customers/assign-by-mobile` 完成前置，然后回到 B/C/P 三端做可见性与结果验证。
- 这不算跳过功能，而是按当前真实产品形态补位。

### 3.3 模板隔离与逐级传递硬规则

以下规则按当前业务确认执行，旧文档若有冲突，以本版为准：

1. 只有上一级模板本身处于 `有效 / 进行中 / 已发布`，才允许向下一级下发。
2. 平台管理员创建并生效的模板，租户管理员可以看到，但继承到租户侧后默认按 `失效` 展示。
3. 租户管理员创建并生效的模板，员工可以看到，但继承到员工侧后默认按 `失效` 展示。
4. 只要租户管理员侧模板处于生效状态，员工就应该可见该模板；模板来源可以是平台创建，也可以是租户创建。
5. 员工侧收到的模板默认都是 `失效`，员工可以手动改状态。
6. 只有员工侧模板状态为 `生效 / 进行中 / 已发布`，其归属客户才可以在 C 端看到。
7. 任何一级若仍为继承态失效，不允许继续向下一级透传。
8. 数据隔离必须同时满足：跨租户不可见、跨员工客户不可见、未归属客户不可见。

## 13. 模板隔离与传递专项验证

这一组是独立门禁，必须覆盖 `平台 -> 租户管理员 -> 员工 -> 客户` 四级链路。以下每条都必须至少在 `知识学习 / 活动中心 / 商城商品 / 商城活动` 四类模板上各执行一轮。

| 用例ID | 场景 | 前置 | 核心动作 | 必须校验 |
|---|---|---|---|---|
| `TPL-001` | 平台模板对租户可见 | 平台管理员创建并生效模板 | 切换到租户管理员查看同模块列表 | 租户管理员可见该模板，且继承状态为 `失效/offline/inactive`，并保留平台来源标记 |
| `TPL-002` | 平台模板未生效不下发 | 平台管理员创建模板但不生效 | 租户管理员查看列表 | 模板不应下发到租户侧 |
| `TPL-003` | 租户模板对员工可见 | 租户管理员创建并生效模板 | 员工进入 B 端对应模块 | 员工可见该模板，但默认状态仍为 `失效/draft/inactive` |
| `TPL-004` | 平台来源模板经租户激活后下发员工 | 平台模板已下发给租户且租户管理员手动改为生效 | 员工进入 B 端对应模块 | 员工能够看到该模板，但仍以员工侧默认失效态接收 |
| `TPL-005` | 租户来源模板经租户激活后下发员工 | 租户管理员创建模板并设为生效 | 员工进入 B 端对应模块 | 员工能够看到该模板，但仍以员工侧默认失效态接收 |
| `TPL-006` | 员工未激活不透传客户 | 员工已收到继承模板但未启用 | 客户打开 C 端对应模块 | 客户不可见该模板 |
| `TPL-007` | 员工激活后客户可见 | 员工手动把继承模板改为生效 | 归属客户打开 C 端对应模块 | 客户可见并可进入详情/参与/兑换 |
| `TPL-008` | 员工回退失效后客户隐藏 | 员工已激活模板且客户可见 | 员工把模板改回失效 | 客户刷新后不可再见该模板 |
| `TPL-009` | 同租户员工隔离 | 员工 A/B 同属一租户 | 仅员工 A 启用模板，分别用 A/B 客户查看 | 仅归属员工 A 的客户可见；员工 B 客户不可见 |
| `TPL-010` | 跨租户隔离 | 租户 A/B 各有模板 | 分别用 A/B 租户管理员、员工、客户查看 | 模板不跨租户透传，任何一级都不能看到对方租户数据 |

执行要求：

- `TPL-*` 每条至少覆盖四类模板：学习、活动、商城商品、商城活动。
- 任何“可见”都要同时验证“状态是否仍为默认失效”和“是否允许继续向下透传”。
- 员工侧如果页面展示来源标识，允许是 `公司模板/个人模板` 这类前端文案，但隔离与下发判断必须以可见范围和状态为准。

## 14. 积分专项验证

这一组不是“积分规则存在即可”，而是要验证 `发分 -> 扣分 -> 余额 -> 流水 -> B 端客户档案 -> P 端规则` 全链路一致。

| 用例ID | 场景 | 前置 | 核心动作 | 必须校验 |
|---|---|---|---|---|
| `POINTS-STAT-001` | 签到发分 | 在 P 端把 `signInPoints` 改成唯一测试值，如 `11` | 客户 A 在 C 端完成签到 | C 端提示成功；`/api/points/summary` 增加 `11`；`/api/points/transactions` 最新流水为签到；B 端客户积分流水出现同一条记录 |
| `POINTS-STAT-002` | 签到防重 | 同一客户当天已完成一次签到 | 再次点击签到 | UI 显示 `今日已签到`；余额不增加；流水不新增第二条签到记录 |
| `POINTS-STAT-003` | 新客户实名奖励 | 在 P 端把 `newCustomerVerifyPoints` 改成唯一测试值，如 `211` | 新手机号首次实名 | 新客户到账 `211`；二次登录/刷新不会重复发分；B 端客户档案出现实名后积分记录 |
| `POINTS-STAT-004` | 分享实名奖励 | 在 P 端把 `customerShareIdentifyPoints` 改成唯一测试值，如 `51` | 客户 A 分享，客户 B 打开分享页并实名 | 奖励发给分享人 A；B 如为首实名仅命中新客户实名规则；重复打开分享不重复奖励 |
| `POINTS-STAT-005` | 学习完课发分 | B 端创建带奖励积分的学习内容，如 `31` 分 | 客户 A 完课 | C 端积分增加 `31`；学习记录可见；B 端学习效果客户列表包含 A |
| `POINTS-STAT-006` | 活动完成发分 | B 端创建带奖励积分的活动，如 `41` 分 | 客户 A 完成活动 | C 端积分增加 `41`；活动记录可见；B 端活动效果客户列表包含 A |
| `POINTS-STAT-007` | 商城商品兑换扣分 | B 端创建消耗积分商品，如 `60` 分 | 客户 A 发起兑换 | C 端余额减少 `60`；`/api/points/detail` 余额同步下降；我的兑换出现记录；B 端订单产生 |
| `POINTS-STAT-008` | 商城活动参与发分 | B 端创建商城活动奖励，如 `25` 分 | 客户 A 参与商城活动 | C 端积分增加 `25`；B 端商城效果回流；流水类型正确 |
| `POINTS-STAT-009` | 积分汇总一致性 | 已完成签到/实名/学习/活动/兑换至少各一条 | 分别打开首页积分入口、个人中心积分卡、积分明细、我的兑换 | 四处余额、流水、兑换记录一致，不允许只一处更新 |

执行要求：

- 每个积分动作都先记基线值，再做动作，再比增量。
- 积分规则必须使用唯一测试值，避免被种子数据干扰。
- 任何积分链路都要验证“无重复发分、无重复扣分、无余额倒挂”。

## 15. 指标专项验证

这一组必须用“基线差值法”做，不接受只看图表有没有渲染。

| 用例ID | 场景 | 前置 | 核心动作 | 必须校验 |
|---|---|---|---|---|
| `METRIC-001` | P 端指标配置口径 | 打开 P 端 `指标配置` | 搜索并打开 `c:C端日活 (DAU)`、`c:签到率`、`c:积分兑换率`、`c:30天登录次数(C端)`、`c:30天签到次数(C端)` | 口径、数据表、计算方式说明完整，无空白占位文案 |

| `METRIC-003` | 今日签到客户数增量 | 记录 B 端工作台基线 | 客户 A 完成签到 | `今日签到客户数` 增量正确；钻取列表包含客户 A；重复签到不再增加 |

## 16. 标签专项验证

标签不能只验“能创建”。必须覆盖 `标签定义 -> 规则命中 -> 任务执行 -> 作业日志 -> B 端客户画像/筛选`。

| 用例ID | 场景 | 前置 | 核心动作 | 必须校验 |
|---|---|---|---|---|
| `TAG-001` | 标签列表 CRUD | 打开 P 端标签列表 | 创建、编辑、停用、删除测试标签 | 列表状态正确，删除后不残留脏引用 |
| `TAG-002` | 固定值规则命中 | 先创建目标标签，如 `高价值客户` | 在 `标签规则库` 用 `const` 模式创建规则并执行 | 作业日志出现命中客户与固定 `outputValue`；B 端客户画像展示目标标签 |
| `TAG-003` | 映射值规则命中 | 先创建目标标签集 | 在 `标签规则库` 用 `map` 模式按 `annual_income` 或 `sign_count_30d` 配映射规则 | 命中客户的 `outputValue` 与映射条件一致，不是固定值乱写 |
| `TAG-004` | 规则优先级冲突 | 为同一客户配置两条都能命中的规则 | 规则优先级分别设为 `10` 和 `100`，执行任务 | 最终生效结果以高优先级规则为准；日志可追溯 |
| `TAG-005` | 规则停用后失效 | 已有一条生效规则 | 将规则停用后再次执行任务 | 不再产生新的命中结果；状态与日志一致 |
| `TAG-006` | 手动执行与日志查看 | 标签规则已存在 | 点击 `执行`，再查看任务日志 | 任务状态、规则数、客户数、日志明细四项完整 |
| `TAG-007` | B 端标签库与自定义标签 | 打开 B 端编辑标签弹层 | 查看推荐标签、分组标签，创建一条自定义标签并绑定客户 A | `recommended/groups/custom` 数据完整；客户详情页可见新标签 |
| `TAG-008` | 标签筛选回流 | 客户 A 已命中或绑定目标标签 | 在 B 端客户库按标签筛选 | 客户列表能准确命中客户 A，无误召回其他客户 |

补充说明：

- P 端标签规则当前支持 `const` 与 `map` 两种输出模式，两种都必须测。
- 规则条件至少覆盖一条 `客户属性` 指标和一条 `客户行为` 指标。
- B 端不仅要看客户详情页，还要验证标签筛选、编辑标签面板、标签库接口三处一致。
