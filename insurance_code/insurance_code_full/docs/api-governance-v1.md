# insurance_code API治理方案（v1）

更新时间：2026-02-23
治理目标：保证C端接口稳定可联调，避免契约漂移。

## 1. 契约冻结与单一事实源

- 冻结文件：`./openapi-c-v1.yaml`
- 规则：
  - 不删除、不重命名、不改变字段语义。
  - 仅允许“向后兼容新增”字段（默认可选）。
  - 字段命名保持现状（如`is_verified_basic`）。

## 2. 版本策略

- 当前版本：`v1`（路径维持现有`/api/*`，由文档标注版本）。
- 兼容变更：新增可选字段/新增接口。
- 破坏性变更：升级`v2`，并保留`v1`至少一个发布周期。

## 3. 错误码标准

统一响应：
```json
{ "code": "ERROR_CODE", "message": "错误信息" }
```

### 3.1 HTTP状态码映射
- `400` 参数错误：`INVALID_*`、`CODE_EXPIRED`
- `401` 未认证：`UNAUTHORIZED`
- `403` 权限/实名门槛：`NEED_BASIC_VERIFY`
- `404` 资源不存在：`*_NOT_FOUND`
- `409` 业务冲突：`ALREADY_*`、`INSUFFICIENT_POINTS`、`OUT_OF_STOCK`
- `410` 资源过期：`TOKEN_EXPIRED`
- `429` 频控：`SMS_LIMIT_REACHED`
- `500` 系统异常：`INTERNAL_ERROR`

## 4. 变更流程（必须执行）

1. 提交API变更提案（模板：背景/影响/兼容性/回滚）。
2. 架构评审（后端+前端+测试+产品）。
3. 更新OpenAPI与示例。
4. CI执行契约检查、兼容性检查。
5. 合并后发布变更日志。

负责人建议：
- 契约Owner：后端负责人
- 兼容性Owner：前端负责人
- 变更审批：架构师

## 5. CI治理门禁

- `openapi lint`（Spectral）必须通过。
- 合约测试（Dredd/Schemathesis任选）必须通过。
- 对比主干OpenAPI差异：出现破坏性变更自动失败。
- 关键接口必须有示例请求/响应快照。

## 6. 联调纪律

- 每日固定窗口发布staging（建议18:00前）。
- 变更提前24小时公告到联调群。
- 非紧急变更不允许当日直发prod。

## 7. 关键决策与取舍

- 决策：以OpenAPI文件作为唯一事实源，代码实现服从契约。
- 取舍：接口迭代灵活性下降，但联调稳定性显著提升。
- 决策：v1阶段禁止破坏性变更，破坏性需求进入v2。
- 取舍：短期可能引入兼容层代码，但避免前端反复返工。

## 8. 落地实施步骤与负责人建议

1. 建立API变更模板并纳入PR检查（后端负责人）。
2. CI加入openapi lint与diff breaking检测（DevOps负责人）。
3. 每个接口补齐示例请求/响应（后端负责人）。
4. 前端基于契约生成类型并做每日回归（前端负责人）。
5. 每周一次契约评审会，处理新增需求（架构师主持）。
