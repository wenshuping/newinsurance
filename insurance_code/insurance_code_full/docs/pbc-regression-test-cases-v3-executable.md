# P/B/C 全链路可执行测试用例（v3，细粒度）

更新时间：2026-02-28
适用版本：`server/skeleton-c-v1` + `insurance_code_P` + `insurance_code_B` + `insurance_code`

## 1. 目标与范围

本用例用于验证以下高风险链路是否真实可用且数据准确：

1. 租户创建 -> 员工创建 -> 客户分配 -> B端配置模板 -> C端可见。
2. 客户归属与数据权限隔离（租户、团队、角色）。
3. 事件上报（v1兼容）是否入库且可追溯。
4. 指标计算口径是否准确（含累计登录天数、累计签到天数、分享次数）。
5. 标签规则执行（固定值/映射值）是否按预期写入客户标签。
6. 回归防护：修复点不反向引入默认客户/默认全量分配等问题。

## 2. 测试环境

- API：`http://127.0.0.1:4000`
- C端：`http://localhost:3000`
- B端：`http://localhost:3001`
- P端：`http://localhost:3002`
- 平台管理员账号：`platform001 / 123456`

说明：
- 本项目当前为 `STORAGE_BACKEND=postgres`，测试时禁止用“离线脚本直接改内存”替代在线 API 结果判断。
- 客户分配优先使用在线接口 `POST /api/p/customers/assign-by-mobile`。

## 3. 角色矩阵

- 平台管理员（P）：可创建租户、跨租户管理、分配客户。
- 公司管理员（P/B）：可管理本租户员工、客户、模板。
- 团队主管（B）：可看本团队客户（含组员客户）。
- 业务员（B）：默认无客户，仅看分配给自己的客户。
- 客户（C）：仅看其归属业务员创建并生效的内容模板。

## 4. 前置数据规范

每次执行用例，创建一套唯一测试数据（时间戳后缀）：

- 租户：`回归租户_<ts>`
- 公司管理员：`tenant_<ts>@demo.local`
- 业务员A：`agent_a_<ts>@demo.local`
- 业务员B：`agent_b_<ts>@demo.local`
- 客户A：`139<8位>`
- 客户B：`138<8位>`

## 5. 主链路用例（必须全部通过）

### CASE-FLOW-001 创建租户与管理员

步骤：
1. P端登录 `platform001`。
2. 进入租户管理创建租户，填写租户名、管理员邮箱、初始密码。
3. 用新管理员账号登录P端。

预期：
- 租户列表出现新租户。
- 管理员登录成功，`session.tenantId = 新租户ID`。

校验接口：
- `POST /api/p/tenants`
- `POST /api/p/auth/login`

### CASE-FLOW-002 新增B端员工默认无客户

步骤：
1. 公司管理员进入员工管理，创建业务员A。
2. 用业务员A登录B端。
3. 打开客户列表。

预期：
- 客户列表为空（0条）。
- 不出现默认mock客户。

校验接口：
- `GET /api/b/customers` 返回 `list=[]`。

### CASE-FLOW-003 客户注册前不可分配

步骤：
1. 调用 `POST /api/p/customers/assign-by-mobile`，mobile使用未注册手机号。

预期：
- 返回 `404 CUSTOMER_NOT_FOUND`。

### CASE-FLOW-004 客户注册后分配

步骤：
1. 客户A在C端走一次登录验证（`verify-basic`）。
2. 公司管理员将客户A分配给业务员A（assign-by-mobile）。
3. 业务员A在B端刷新客户列表。

预期：
- 分配接口成功。
- B端客户列表出现客户A。

### CASE-FLOW-005 业务员创建三类模板

步骤：
1. 业务员A在B端创建学习资料（published）。
2. 创建活动模板（online）。
3. 创建积分商城商品（active）。

预期：
- 三类模板创建成功，`createdBy = 业务员A`，`tenantId = 当前租户`。

### CASE-FLOW-006 C端可见性验证

步骤：
1. 用客户A登录C端。
2. 打开知识学习、活动中心、积分商城。

预期：
- 能看到业务员A创建的学习资料/活动/商品。
- 看不到其他业务员或其他租户模板。

## 6. 权限与隔离用例

### CASE-AUTH-001 system-assign 禁止空列表全量分配

步骤：
1. 调用 `POST /api/p/customers/system-assign`，传 `customerIds=[]`。

预期：
- 返回 `400 CUSTOMER_IDS_REQUIRED`。

### CASE-AUTH-002 跨租户隔离

步骤：
1. 在租户X创建模板。
2. 用租户Y客户登录C端查看。

预期：
- 租户Y客户看不到租户X模板。

### CASE-AUTH-003 角色可见范围

步骤：
1. 同租户下创建公司管理员、团队主管、业务员。
2. 分配不同团队/归属客户。

预期：
- 公司管理员：可看租户内全部客户。
- 团队主管：可看本团队客户。
- 业务员：仅看owner为自己的客户。

## 7. 事件埋点测试（v1兼容）

### CASE-EVT-001 C端关键事件入库

操作路径：
1. C端登录。
2. 打开知识学习列表、进入课程详情。
3. 打开活动列表与活动详情。
4. 打开积分商城列表与商品详情并发起兑换。

预期：
- `p_track_events` 有对应事件（例如 `c_learning_*`, `c_activity_*`, `c_mall_*`）。
- 事件包含 tenant/actor/path/source 基础字段。

### CASE-EVT-002 分享分端统计

步骤：
1. 触发C端分享、B端分享。
2. 查询事件表。

预期：
- C/B分享事件分开记录，不混算。
- 可按人、按日统计次数。

## 8. 指标准确性测试

### CASE-MET-001 累计登录天数

口径：客户历史“登录成功”去重自然日总数。

步骤：
1. 连续2天触发客户登录事件（可通过改测试日模拟）。
2. 查询指标接口或聚合结果。

预期：
- 值=2。

### CASE-MET-002 累计签到天数

口径：客户历史“签到成功”去重自然日总数。

步骤：
1. 连续N天签到（或造数N天）。
2. 查询指标。

预期：
- 值=N。

### CASE-MET-003 分享次数（按端/按人/按天）

步骤：
1. 同一客户同一天触发C端分享3次、B端分享2次。
2. 查询指标统计。

预期：
- `c_share_daily_count = 3`
- `b_share_daily_count = 2`
- 不存在跨端合并误差。

### CASE-MET-004 指标口径备注一致性

步骤：
1. 在P端指标配置查看“备注/口径说明”。
2. 对照 `metric-definition-v1.md`。

预期：
- 展示口径与文档一致，不出现“仅注入字符串不生效”。

## 9. 标签系统测试

### CASE-TAG-001 标签定义CRUD

步骤：
1. 新建标签（枚举型/数值型）。
2. 编辑、停用、启用。

预期：
- 列表状态与详情一致。

### CASE-TAG-002 规则固定值

步骤：
1. 建规则：IF（条件）THEN（目标标签=固定值“高价值”）。
2. 触发命中并执行任务。

预期：
- 客户标签值写入“高价值”。

### CASE-TAG-003 规则映射值

步骤：
1. 建规则：映射来源字段（例如收入区间）到标签枚举值。
2. 执行规则任务。

预期：
- 映射命中准确，默认值兜底有效。

### CASE-TAG-004 多规则优先级

步骤：
1. 同标签配置两条规则（P0、P1）均可命中。
2. 执行任务。

预期：
- 仅输出优先级最高规则结果。

### CASE-TAG-005 B端客户画像读取

步骤：
1. 在B端客户详情查看标签结果。

预期：
- 显示与规则执行结果一致。

## 10. 数据一致性校验（建议每轮必做）

- API层校验：
1. `GET /api/b/customers`
2. `GET /api/learning/courses`
3. `GET /api/activities`
4. `GET /api/mall/items`

- DB层校验（通过项目Node脚本/SQL）：
1. 客户归属：`c_customers.owner_agent_id` 是否与分配一致。
2. 模板归属：`p_learning_materials/p_activities/p_products.created_by` 是否等于创建员工ID。
3. 事件链路：`p_track_events` 是否覆盖关键动作。
4. 指标聚合：`p_metric_*` 与事件明细抽样核对。
5. 标签结果：标签值表与规则任务日志一致。

## 11. 非功能与异常场景

### CASE-NFR-001 并发分配

步骤：
1. 10并发请求分配同一客户到同一员工。

预期：
- 最终归属稳定，无脏写；审计日志完整。

### CASE-NFR-002 幂等性

步骤：
1. 同一请求重复提交（活动完成、积分扣减、分配）。

预期：
- 无重复扣分/重复订单/重复分配副作用。

### CASE-NFR-003 重启恢复

步骤：
1. 完成分配与模板创建后重启API。
2. 重新登录B/C端查看。

预期：
- 数据不丢失，权限和可见性不变化。

## 12. 出口标准（Go/No-Go）

必须全部满足：
1. 主链路 6/6 通过。
2. 权限隔离用例 3/3 通过。
3. 事件、指标、标签核心用例全部通过。
4. 无“新员工默认有客户”“空列表默认全量分配”回归。
5. 重启后数据一致。
