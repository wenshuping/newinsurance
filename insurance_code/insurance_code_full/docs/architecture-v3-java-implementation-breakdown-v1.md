# 保云链 Java 架构落地拆解 v1（基于 v3.1）

更新时间：2026-02-28  
适用范围：P/B/C 三端联调、Node -> Java 渐进迁移

---

## 1. 执行原则

1. 兼容优先：先兼容 v1 接口与字段，不做破坏性变更。
2. 风险优先：先做租户隔离、权限默认拒绝、鉴权链路。
3. 可回滚：每阶段都能回切 Node 稳定路径。
4. 可验证：每阶段必须有自动化测试和浏览器回归结果。

---

## 2. 角色与责任

1. 架构负责人（你/Tech Lead）
- 决策边界、评审接口契约、Go/No-Go。

2. 后端 A（Identity）
- 租户、团队、员工、登录、权限。

3. 后端 B（Engagement）
- 客户、学习、活动、商城、积分。

4. 后端 C（Intelligence）
- 事件、指标、标签与规则。

5. 前端负责人
- SDK 对接、兼容字段、P/B/C 冒烟联调。

6. 测试负责人
- 回归用例执行、权限/数据隔离验证、口径核对。

7. DevOps
- CI/CD、容器、监控、日志、灰度策略。

---

## 3. 里程碑与周计划（8 周 MVP）

## 第 1 周：契约冻结与基线治理

目标：冻结 v1 契约、建立统一工程骨架。  
负责人：架构负责人 + 后端 A + 前端负责人

任务：
1. 冻结 `OpenAPI v1`（P/B/C 全接口）。
2. 建 `shared-contracts`（DTO、错误码、枚举）。
3. 建 Java 多模块骨架：`identity-service`、`engagement-service`、`intelligence-service`、`gateway`。
4. 建代码质量门禁：Checkstyle、SpotBugs、PMD、Jacoco。
5. 建 CI 流水线（编译、单测、合约校验）。

产出：
1. 契约基线文档。
2. Java 工程可编译可启动。
3. CI 首次全绿。

验收：
1. 前端 SDK 不改业务代码即可连到旧接口。
2. Java skeleton 启动健康检查通过。

---

## 第 2 周：租户上下文与鉴权链路

目标：实现“无默认兜底”的认证与租户上下文。  
负责人：后端 A

任务：
1. 网关实现 JWT 校验与 claim 透传。
2. 实现 TenantContext（tenantId/orgId/teamId/userId/role）。
3. 缺失 claim 统一 401/403，禁止 tenant=1 兜底。
4. 实现统一异常处理与 requestId。
5. 登录/刷新 token 接口 Java 化（兼容返回结构）。

产出：
1. `identity-service` 登录能力。
2. 网关鉴权和上下文中间件。

验收：
1. 未登录访问受控接口全部被拒绝。
2. claim 缺失请求 100% 拦截。

---

## 第 3 周：权限默认拒绝 + 团队模型

目标：权限系统切到“默认 false”。  
负责人：后端 A + 前端负责人

任务：
1. 建权限配置表，默认 `enabled=false`。
2. 实现角色-页面-动作权限校验。
3. 实现数据范围 `SELF/TEAM/TENANT/ALL`。
4. 团队管理支持：新增、改名、删除（仅空团队可删）。
5. 前端接入权限接口，未授权页面隐藏/禁用。

产出：
1. 权限管理 API 与缓存失效机制。
2. 团队管理闭环。

验收：
1. 新租户未配置权限时，所有受控页面不可操作。
2. 公司管理员可看租户全量，团队长可看团队范围。

---

## 第 4 周：客户归属与 C 端可见性（核心链路）

目标：确保“C 端内容跟随 B 端业务员生效配置”。  
负责人：后端 B + 测试负责人

任务：
1. 客户归属模型：`owner_user_id`、`team_id`、`tenant_id`。
2. 新增业务员默认不分配客户。
3. 提供分配脚本与 API（批量按手机号分配）。
4. C 端内容查询按客户归属过滤（学习/活动/商城）。
5. 加入可见性调试接口（仅测试环境）。

产出：
1. 客户分配脚本。
2. C 端三大模块可见性规则落地。

验收：
1. 同租户不同业务员客户看到的内容不同且正确。
2. 新员工未分配客户时，C 端不会自动出现客户内容。

---

## 第 5 周：事件定义与埋点链路

目标：打通事件定义 -> 上报 -> 轨迹展示。  
负责人：后端 C + 前端负责人

任务：
1. 实现事件定义管理（P 端事件管理菜单）。
2. 支持 v1 上报格式兼容。
3. 增加分享事件拆分：`c_share_success`、`b_share_success`。
4. 行为轨迹展示中文事件名和业务对象（课程名/活动名/商品名）。
5. 轨迹支持按天筛选与折叠展开。

产出：
1. 事件字典与事件日志查询 API。
2. B 端客户详情轨迹可读化。

验收：
1. “知识学习”入口行为可在轨迹中稳定显示。
2. C/B 分享事件分开统计并可追溯。

---

## 第 6 周：指标平台与口径同步

目标：指标定义、计算、展示一致。  
负责人：后端 C

任务：
1. 指标定义管理（P 端）。
2. 增加指标：
- 单客户累计登录天数
- 单客户累计签到天数
- 单客户 C 端分享次数（日/总）
- 单客户 B 端分享次数（日/总）
3. 建日聚合任务（D+0 / D+1 修正）。
4. 指标配置备注与指标口径同步展示。
5. 对账任务：事件日志 vs 指标值。

产出：
1. 指标计算作业。
2. 对账报表。

验收：
1. 指标误差率 <= 0.1%。
2. 轨迹抽样与指标抽样一致。

---

## 第 7 周：标签列表与规则库联调

目标：标签系统按 PRD 可用。  
负责人：后端 C + 前端负责人

任务：
1. 标签定义 CRUD。
2. 规则库支持条件组：客户属性 + 客户行为指标。
3. THEN 支持 `固定值` 与 `映射值`：
- 固定值：写死标签值
- 映射值：按规则映射表动态赋值
4. 客户属性扩展（年龄、性别、收入等）
5. 规则试跑与命中预览。

产出：
1. 标签规则执行引擎。
2. 命中记录与回溯查询。

验收：
1. 标签命中可解释（命中条件、原始值、目标值）。
2. THEN 选择标签后前端正确展示已选标签。

---

## 第 8 周：全链路回归 + 灰度准备

## 6. 测试矩阵（重点）

## 6.1 数据权限

1. 跨租户读取拦截。
2. 团队长仅团队数据。
3. 业务员仅本人客户数据。
4. 公司管理员可见租户全量。

## 6.2 数据正确性

1. 事件上报量 = 轨迹记录量（抽样）。
2. 指标值与事件聚合一致。
3. 标签命中结果与规则条件一致。
4. C 端内容可见性与客户归属一致。

## 6.3 角色权限

1. 未授权用户无法创建活动/商品/内容。
2. 页面权限未配置默认不可操作。
3. 权限变更后 5 分钟内生效（缓存失效验证）。

## 6.4 安全

1. Token 缺失/过期/伪造拦截。
2. CSRF（若 Cookie 模式）拦截。
3. 敏感接口二次确认。
4. 审计日志留痕完整。

---

## 7. 上线门禁（Go/No-Go）

1. 自动化测试通过率 100%。
2. 核心链路手工回归全部通过。
3. 越权测试 0 漏洞。
4. 口径对账（事件/指标/标签）误差 <= 0.1%。
5. 预发稳定运行 72 小时。

## 9. 风险登记（首版）

1. 迁移期间双写/双读逻辑复杂导致口径偏差。
2. 历史测试数据污染影响回归结果。
3. 角色权限模型与前端菜单模型不一致。
4. 业务高峰时 Kafka 消费滞后导致指标延迟。

对应缓解：
1. 对账任务 + 阈值告警 + 回放机制。
2. 标准化测试夹具与数据隔离环境。
3. 统一权限枚举与前后端契约。
4. 消费分区扩容 + 死信处理 + 延迟告警。
