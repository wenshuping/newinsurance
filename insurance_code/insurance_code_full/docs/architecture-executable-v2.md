## 1. 目标与约束

> 执行线说明：本文件对应当前生效执行线（Node 模块化单体）。Java 三服务方案属于迁移规划，不作为当前迭代交付目标。

### 1.1 目标
- 在不影响 P/B/C 现有联调的前提下，将当前原型后端演进为可上线架构。
- 保障多租户隔离、权限正确性、事件/指标/标签口径一致性。
- 提供可验证、可回归、可灰度的迁移路径。

### 1.2 约束
- 前端接口协议尽量保持兼容（先兼容再优化）。
- 不允许默认身份兜底（禁止隐式 tenant/org/team/actor）。
- 新增能力必须能通过自动化回归与浏览器回归。

## 2. 当前基线（As-Is）

- 后端：Node.js + Express（单体，`server/skeleton-c-v1`）
- 数据：PostgreSQL（默认）+ JSON file 兜底
- 权限：RBAC + 数据范围过滤（tenant/team/owner）
- 已有模块：租户、员工、客户、活动、学习、商城、积分、埋点、指标、标签规则

主要风险：
- 业务逻辑集中在单体进程，模块边界弱。
- 个别历史逻辑仍存在“演示态”代码路径。
- 事件->指标->标签链路缺统一版本化治理。

## 3. 目标架构（To-Be）

### 3.1 近期目标（3-6周）
- 保持单体部署形态不变，先做“模块化单体”。
- 抽离清晰领域模块：
  - IAM（认证鉴权/租户/角色）
  - CRM（客户/归属/档案）
  - Content（学习资料）
  - Campaign（活动）
  - Commerce（商城/兑换/核销）
  - Event & Metrics（埋点/聚合）
  - Tag Engine（规则打标）

### 3.2 中期目标（6-12周）
- 拆为 3 个服务组：
  - `identity-service`（IAM + tenant context）
  - `engagement-service`（内容/活动/商城/积分）
  - `data-intelligence-service`（事件/指标/标签）
- 引入 API Gateway（统一鉴权、审计、限流）。

### 3.3 长期目标（12周+）
- 按业务流量和团队边界进一步拆分（可选）。
- 引入消息总线与异步任务编排，建设数据中台与统一指标平台。

## 4. 技术栈定版（建议）

### 4.1 应用层
- Runtime：Node.js 22 LTS
- Framework：Express（当前保留）/ 后续可平滑迁移到 NestJS
- API：REST + OpenAPI 3.1（契约优先）

### 4.2 数据层
- 主库：PostgreSQL 16
- 缓存：Redis 7
- 对象存储：MinIO / 云 OSS
- 迁移：SQL migration（版本化）+ 数据回填脚本

### 4.3 运维与质量
- 观测：Prometheus + Grafana + Loki（或 ELK）
- 链路追踪：OpenTelemetry
- CI/CD：分支质量门禁（lint/test/e2e/regression）

## 5. 多租户与权限强约束

### 5.1 硬性规则
- 所有业务表必须包含 `tenant_id`。
- 所有查询必须显式注入 `tenant_id` 条件。
- 缺失 tenant context 一律拒绝（HTTP 400/401/403），不得兜底到 tenant=1。

### 5.2 数据范围策略
- 平台管理员：可跨租户（仅平台域接口）。
- 公司管理员：仅本租户全量。
- 团队主管：仅本团队。
- 业务员：仅本人客户（owner_user_id）。
- 客户：仅本人 + 归属业务员可见内容。

### 5.3 审计
- 敏感操作（权限、导出、退款、人工改分）强制写审计日志。
- 审计保留 >= 180 天。

## 6. 事件-指标-标签一体化标准

### 6.1 事件
- 事件字典版本化（v1/v2...），字段包含：事件名、来源端、口径、属性、去重键。
- C/B 分享事件分端统计，不合并口径。

### 6.2 指标
- 指标必须绑定口径定义（SQL/聚合逻辑+时间窗口+去重规则）。
- 关键指标（如累计登录天数、累计签到天数）必须有回归校验样本。

### 6.3 标签
- 规则引擎支持 `const` 与 `map` 两种输出模式。
- 标签删除保护：被规则引用时禁止删除（409）。

## 7. 数据模型补齐（必须执行）

### 7.1 必要唯一约束
- 用户与员工账号：`(tenant_id, email)`、`(tenant_id, mobile)`
- 幂等：`(tenant_id, biz_type, biz_key)`
- 指标日聚合：`(tenant_id, stat_date, metric_key, actor_id)`

### 7.2 事务一致性
- 兑换链路：扣分、扣库存、生成核销记录必须同事务或可靠补偿。
- 幂等冲突返回稳定错误码。

## 8. 分阶段实施计划

### Phase A（第1-2周）
- 清理默认兜底身份逻辑。
- 固化 tenant context 中间件。
- 完成权限矩阵自动化测试。

### Phase B（第3-4周）
- 统一事件字典与指标口径。
- 打通标签规则执行日志与校验用例。
- 完成 P/B/C 回归自动化脚本（当前已有基础版本）。

### Phase C（第5-6周）
- 抽象领域服务层，降低路由耦合。
- 引入消息队列处理异步任务（对账、重算、补偿）。
- 上线前压测与故障演练。

## 9. 验收标准（DoD）

### 9.1 功能
- P/B/C 核心链路回归通过率 100%。
- 租户、团队、人员、客户归属链路全部可复现。

### 9.2 数据正确性
- 事件、指标、标签口径一致；抽样校验误差 <= 0.1%。
- 跨角色可见范围符合矩阵定义。

### 9.3 稳定性
- 关键接口 P95：
  - 登录/列表 <= 800ms
  - 签到/兑换/核销 <= 500ms
- 核心链路成功率 >= 99.5%。
